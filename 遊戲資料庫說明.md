# 遊戲資料庫使用方式

可編輯的物品、NPC 與地點資料集中在 `src/_GameDatabase.twee`，不會隨著存檔變動；玩家背包與 NPC 關係等每局不同的內容，則在 `StoryInit` 初始化為存檔資料。

## ID、翻譯與資料分工

- `tea`、`alice` 等是固定 ID。發行存檔後不要改 ID，才能避免舊存檔壞掉。
- 物品與 NPC 的顯示名稱、說明都放在 `locales["zh-TW"]` 與 `locales.en`。新增語言時，複製 `en` 區塊、填入翻譯，再把玩家的 `$language` 改為語言代碼即可。
- 物品的價格、效果、分類與 `tags` 放在 `items`；不要把顯示文字寫回遊戲段落。

## 劇情事件與任務

目前不使用事件或任務資料庫。搶匪、看電視、健身等劇情與分支，直接寫在各自的 `.twee` 場景中，這是大型 Twine 遊戲也常用的做法。

日後翻譯時，先直接翻譯這些 `.twee` 內文即可；等劇情與語言數量真的變多，再逐步把重複的短句抽到語言資料表，避免一開始的編輯流程太複雜。

## 正式場所與工作的衛生規則

`setup.gameDatabase.venueRules` 是「場所的共用入場規則」。目前咖啡廳已套用；日後新增飯店、辦公室或其他工作場所時，複製 `cafe` 的格式即可。

```js
hotel: {
    name: "飯店",
    hygiene: {
        entry: {
            minimum: 50,
            message: "服務人員委婉地請你先整理儀容。",
            returnPassage: "市區",
            returnLabel: "返回市區"
        },
        work: {
            minimum: 50,
            message: "主管請你整理好儀容後再來打卡。",
            returnPassage: "飯店大廳",
            returnLabel: "返回大廳"
        }
    }
}
```

- `minimum: 20` 的意思是衛生 **低於 20** 才拒絕，20 本身可以通過。
- `entry` 管正式場所入口；`work` 管開始或繼續工作。兩者可設定不同門檻與文字。
- 入口先用 `setup.canUseVenue("hotel", "entry")` 檢查；工作按鈕使用 `<<venueGo "開始工作" "hotel" "work" "飯店工作">>`。
- 工作 passage 本身也應再檢查一次，避免舊歷史紀錄或其他事件直接跳轉繞過規則；可仿照「在咖啡廳工作」開頭。

## 開放時間、開鎖與警衛規則

若場所白天可正常進入、關門後可嘗試開鎖，請在同一個場所資料加入 `access`。大學是完整範例；開鎖共用流程在 `src/VenueAccess.twee`，通常不需要修改。

```js
access: {
    entryPassage: "大學",
    returnPassage: "市區",
    returnLabel: "返回市區",
    openDays: [1, 2, 3, 4, 5],
    openHour: 7,
    closeHour: 19,
    lockpick: { difficulty: 15, /* 其他數字請複製大學範例 */ }
}
```

- 星期日是 `0`、星期一到星期五是 `1`～`5`、星期六是 `6`。
- 入口按鈕使用 `<<venueEnter "🏫大學" "university" 3>>`；最後的 `3` 是前往該場所花的分鐘數。
- 開鎖成功率依「基礎值 + 靈巧等級加成 + 有效幸運加成 - 難度」計算；每個場所可設定自己的 `difficulty`。
- 開鎖失敗但未被警衛發現時，該場所冷卻 60 分鐘；被發現時扣聲望，並只封鎖該場所到當天結束。
- `successMessage`、`failedMessage`、`caughtMessage`、`lockoutMessage` 與 `cooldownMessage` 都能依場所改成專屬事件文字。

## 角色性別與大學科系

新遊戲按下開始後，會先選擇 `$playerGender`，再選擇 `$majorId`；確認頁的「我再想一下」可回到選系畫面。可編輯選項集中在 `setup.gameDatabase.characterGenders` 與 `setup.gameDatabase.majors`。

每個科系只要指定一個 `majorSkill` 與一個 `minorSkill`：

```js
example_major: {
    name: "範例科系",
    majorSkill: "tech",
    minorSkill: "knowledge",
    description: "這裡是選系畫面的簡短介紹。"
}
```

大學課程的時間、單堂學費、主修與副修經驗都在 `venueRules.university.education`。目前為 60 分鐘、30 元、主修 +12 XP、副修 +5 XP。兩項技能都滿級時不能再上課，也不會收費。

入學滿 7 天後，可到大學的「學生服務中心」免費轉系。可轉系的天數由 `majorChangeWaitDays` 控制，實際資格由 `setup.canChangeMajor()` 判定；日後要加考試、聲望或費用，只需要擴充這個函式，不必修改選系畫面。

## 公園休息與角色資訊

- 公園休息的分鐘數與恢復量在 `setup.balance.parkRest`。目前為 15 分鐘、體力 +1、心情 +2；它仍會正常推進時間，所以不會是無代價的恢復。
- 左側的「角色」按鈕會開啟獨立角色資訊視窗，顯示性別、科系與文字化聲望。聲望的分段文字集中在 `setup.balance.reputation.tiers`，不需要到介面檔案修改。
- 日後的搶劫、被搶、約會、親密事件與第一次事件紀錄，應先在 `StoryInit` 建立統計變數，再在 `skills.twee` 的 `CharacterSheet` 區塊下方顯示；這樣事件程式與介面會保持分開。
- 左側的五項常駐數值使用 `<<statusBar>>` 共用元件顯示為緊湊進度條：體力／心情為第一列，飢餓／衛生為第二列，慾望獨佔最後一行；金錢與幸運另放在下方。一般數值在 25% 以下變黃、10% 以下變紅；慾望固定為紫色。若日後新增同類數值，直接複製 `StoryCaption` 的一行並換成對應名稱、目前值、最大值與類型即可。
- 進度條與金錢／幸運包在 `status-panel`；其上方預留的頭像空位由 `_UserStyle.twee` 的 `margin-top: 165px` 控制。新增玩家頭像時，可依實際圖片高度調整這個數字。低數值不再顯示額外文字警告，直接以黃／紅進度條提示，讓左側版面保持緊湊。

## 新增物品範例

在 `locales` 的每一種語言各加入 `items.cookie` 的 `name`、`description`、`useMessage`，再加入以下資料：

```js
cookie: {
    icon: "🍪",
    category: "food",
    tags: ["consumable", "food", "shop_711", "gift"],
    price: 35,
    stackable: true,
    usable: true,
    effects: [{ type: "stat", stat: "satiety", amount: 10 }]
}
```

若要在 7-11 販售，再將 `cookie` 加到 `shops.shop_711.itemIds`。背包、購物介面、結帳金額與購物車會自動支援它。

## NPC 資料設計

每個 NPC 有固定資料（`tags`、生日、喜好、事件 ID）與存檔資料（`$npcRelations`）兩部分。可戀愛角色才會建立 `romance`；不可戀愛角色只會有 `affection`。

`alice` 與 `bob` 是依目前規劃文件建立的暫定範例，不代表已設定的正式角色。`alice` 示範可戀愛、忠誠、會在關係事件中敏感反應的角色；`bob` 示範不可戀愛角色。角色設定完成時，直接保留 ID 並替換翻譯、標籤與資料即可。

## TAG 與條件範例

TAG 可同時用於分類和規則。現有 `setup.conditionMet()` 支援以下條件：

```js
{ type: "npcHasTag", npcId: "alice", tag: "loyal" }
{ type: "npcLacksTag", npcId: "bob", tag: "romanceable" }
{ type: "hasItem", itemId: "tea", quantity: 1 }
{ type: "itemHasTag", itemId: "tea", tag: "gift" }
{ type: "relationshipAtLeast", npcId: "alice", stat: "romance", value: 80 }
{ type: "flag", flag: "alice_saw_public_flirting" }
```

日後的事件可先檢查這些條件，再套用結果。例如「具有 `loyal` 標籤的戀人目睹公開曖昧」可扣除其 `romance`；當數值低於角色的 `breakupAtRomance`（目前範例為 40）時，再觸發分手事件。多重交往不需要特別限制，因為每位角色的關係數值是分開儲存的。
