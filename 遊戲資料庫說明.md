# 遊戲資料庫使用方式

可編輯的物品、NPC 與地點資料集中在 `src/_GameDatabase.twee`，不會隨著存檔變動；玩家背包與 NPC 關係等每局不同的內容，則在 `StoryInit` 初始化為存檔資料。

## ID、翻譯與資料分工

- `tea`、`alice` 等是固定 ID。發行存檔後不要改 ID，才能避免舊存檔壞掉。
- 物品與 NPC 的顯示名稱、說明都放在 `locales["zh-TW"]` 與 `locales.en`。新增語言時，複製 `en` 區塊、填入翻譯，再把玩家的 `$language` 改為語言代碼即可。
- 物品的價格、效果、分類與 `tags` 放在 `items`；不要把顯示文字寫回遊戲段落。

## 劇情、分支與翻譯資料庫

現在的劇情資料以「枝語 Story Studio」圖形化編輯器管理；遊戲實際讀取的是編輯器匯出的 `src/_NarrativeDatabase.twee`。JSON 仍是引擎內部的資料格式，但只由工具產生，寫作者不需要手動輸入。Google Sheet 與 `database/story` 的三份 CSV 保留為翻譯協作與批次檢視用途。

日常流程：

1. 開啟「枝語 Story Studio」，按右上角「匯入現有資料」。
2. 第一次可選擇專案裡的 `src/_NarrativeDatabase.twee`；之後也可載入上次下載的 `narrative-project.json`。
3. 依「文本與翻譯 → 選項與結果 → 劇情事件」的順序新增資料。條件與效果直接使用中文表單，不要手寫 JSON。
4. 到「檢查與匯出」，先下載 `narrative-project.json` 作備份。
5. 下載 `_NarrativeDatabase.twee`，覆蓋專案 `src` 裡的同名檔案，再用原本的 Twine 編譯流程重新建置。

編輯器會檢查找不到的文本／選項組、缺少的 ID 和翻譯進度。紅色錯誤修正前不能匯出引擎檔；黃色提醒通常是英文尚未翻譯，不會阻止匯出。完整操作說明在 `story-editor/README.md`，編輯器內也有「新手教學」。

### 完全不懂程式也能照做的舊資料匯入流程

1. 開啟「枝語 Story Studio」網站。
2. 按畫面右上角的「匯入現有資料」。
3. 在檔案選擇視窗依序進入 `D:\New` → `src`。
4. 選擇 `_NarrativeDatabase.twee`，按「開啟」。
5. 畫面上方出現「已匯入 _NarrativeDatabase.twee」就成功了。
6. 左側「劇情事件」會看到 `winley_park_intro` 等事件；「文本與翻譯」會看到 `joel_first_meeting` 等舊文本。
7. 修改時不用按存檔，網站會自動保存在目前瀏覽器；但完成一輪工作後，仍建議到「檢查與匯出」下載「專案備份」。

網站無法直接改動電腦裡的遊戲檔。編輯完成後，請在「檢查與匯出」下載新的 `_NarrativeDatabase.twee`，再把它放進 `D:\New\src`，覆蓋舊的同名檔案。

### 遊戲畫面與資料庫對話的分工

- 遊戲的 Passage 決定「玩家現在要進入哪個場景對話池」。例如溫莉公園入口使用場景 ID `winley_park`。
- 編輯器的「劇情事件」決定這個池子裡有哪些對話、各自何時能出現，以及誰的優先度較高。
- 引擎負責從符合條件的事件中選出實際播放的那一段。因此地圖程式通常不需要寫死 `winley_park_intro`、`winley_park_coffee_first` 等個別事件 ID。

溫莉公園入口目前位於 `src/NPC_Winley.twee`，核心意思是「請引擎從 `winley_park` 場景選一段現在符合條件的劇情」。以後若新增同場景對話，通常只要在 GUI 建立事件並沿用 `winley_park`，不需要再次修改這個 Passage。

### 常用的劇情呼叫語法

只顯示一段文本，不需要條件、選項或效果：

```twine
<<storyText "你的文本_KEY">>
```

從某個場景 ID 自動挑選符合條件、優先度最高的事件。這是多分支遊戲最常用的寫法：

```twine
<<set _pickedEvent to setup.pickNarrativeEvent("你的場景_ID", {})>>
<<if _pickedEvent>>
    <<set $narrativeEventId to _pickedEvent.id>>
    <<set $narrativeContext to {}>>
    <<goto "故事事件">>
<<else>>
    /% 沒有符合條件的事件時不顯示系統提示，繼續留在原場景。 %/
<</if>>
```

確定要直接播放某一個事件 ID，不經過場景挑選：

```twine
<<set $narrativeEventId to "你的事件_ID">>
<<set $narrativeContext to {}>>
<<goto "故事事件">>
```

需要把目前 NPC 傳進事件時：

```twine
<<set $narrativeEventId to "你的事件_ID">>
<<set $narrativeContext to { npcId: "winley" }>>
<<goto "故事事件">>
```

判斷原則：只有固定說明文字才直接用 `<<storyText>>`；有條件、選項、旗標、好感或獎勵時，應呼叫「故事事件」，讓引擎完整執行事件資料。

### 對話 Banner

到 GUI 的「劇情事件」→ 選擇事件 →「基本資料」→「對話 Banner」即可決定該事件使用的橫幅。目前公園尚無專屬圖片，因此溫莉公園事件先使用「街道／公園共用」，實際檔案是 `build/images/street_banner.jpg`。

若未來新增 `park_banner.jpg`：

1. 把圖片放進遊戲編譯後會複製到的 `images` 資料夾。
2. 在公園事件的「對話 Banner」直接輸入 `park`。這個欄位除了既有建議清單，也接受新的 Banner ID，不必修改 GUI 程式。

事件沒有指定 Banner 時，會沿用進入對話前的場景圖片；指定後則以事件的選擇為準。

條件與效果中的一般遊戲變數會讀取 Notion「遊戲變數資料庫」，個別劇情旗標則讀取第二張 Notion「劇情旗標總表（$flags）」，兩者都依分類與型別顯示為下拉選單。例如選擇 `$money` 時，GUI 會自動輸出引擎需要的 `money`。若匯入專案含有尚未登記到 Notion 的旗標，GUI 會另外分組提醒；新增旗標時仍可使用選單末端的自訂入口。

若需要回到舊流程，仍可在匯出頁下載三份 CSV，或使用 Google Sheet 的資料覆蓋 `database/story` 後執行 `node tools/sync-narrative-database.mjs database/story`。

### 事件資料的三層

- `Story Text`：只放顯示文字。`text_key` 是固定 ID；`zh_TW` 與 `en` 是同一段文字的不同翻譯。可在文本中使用 `{{npc:winley}}` 顯示 NPC 名稱，或 `{{value:rentDebt}}` 顯示目前存檔數值。
- `Story Events`：決定「什麼情境可以演什麼」。`priority` 越高越優先；相同 priority 的可用事件會隨機挑選，適合日常對話池。`banner` 則決定該事件上方顯示哪張圖片。
- `Choices`：決定玩家按鈕的順序、顯示文字、效果與下一段。多個事件可以共用同一組選項。

溫莉公園事件已是完整範例：首次見面、三種依次數變化的咖啡日常、散步、好感 30 解鎖日常，全部由三張表選取；搶匪、欠租、昏倒、關係確認等既有事件的劇情文字也已移到 `Story Text`。

可直接沿用的條件：`flagNotSet`、`flagIs`、`flagAtLeast`、`flagAtMost`、`npcAffectionAtLeast`、`stateEquals`、`stateAtLeast`；效果：`setFlag`、`setFlagToday`、`incrementFlag`、`changeNpcAffection`、`changeNpcRomance`、`changeStat`、`gainSkillExp`、`giveItem`、`advanceTime`。這些都已經做成編輯器裡的中文選單；新增前選擇最接近的條件或效果，再依欄位提示填值即可。

### 讓劇情獎勵技能經驗

不要選 `$skillMaxLevel`，那只是全遊戲的技能等級上限。請在 GUI 的效果區按「新增效果」，選「增加技能經驗」，再從 Notion 同步的下拉選單選技能並填入經驗值。

- 玩家一進入事件就得到經驗：加在「劇情事件 → 進場效果」。
- 玩家按下某個選項後才得到經驗：加在「選項與結果 → 選擇後效果」。通常「完成這段劇情才給獎勵」適合放這裡。

此效果會自動處理升級、滿級與健身升級增加體力上限，不要直接修改 `$skills.xxx.exp`。

### 讓 NPC 把物品交給玩家

在最後一個按鈕的「選擇後效果」新增「獲得物品」，輸入物品 ID 與數量。例如物品 ID 填 `ramen`、數量填 `2`，玩家按下按鈕後背包就會增加 2 個泡麵。

物品 ID 必須先存在於 `setup.gameDatabase.items`。目前可用的 ID 包含 `tea`、`weirdDrink`、`ramen`、`cookie`、`recycled_luck_pouch`；輸入不存在的 ID 時不會加入背包。

## 正式場所與工作的衛生規則

`setup.gameDatabase.venueRules` 是「場所的共用入場規則」。目前咖啡廳已套用；日後新增飯店、辦公室或其他工作場所時，複製 `cafe` 的格式即可。

```js
hotel: {
    name: "飯店",
    hygiene: {
        entry: {
            minimum: 50,
            message: "服務人員委婉地請你先整理儀容。",
            returnPassage: "市區",
            returnLabel: "返回市區"
        },
        work: {
            minimum: 50,
            message: "主管請你整理好儀容後再來打卡。",
            returnPassage: "飯店大廳",
            returnLabel: "返回大廳"
        }
    }
}
```

- `minimum: 20` 的意思是衛生 **低於 20** 才拒絕，20 本身可以通過。
- `entry` 管正式場所入口；`work` 管開始或繼續工作。兩者可設定不同門檻與文字。
- 入口先用 `setup.canUseVenue("hotel", "entry")` 檢查；工作按鈕使用 `<<venueGo "開始工作" "hotel" "work" "飯店工作">>`。
- 工作 passage 本身也應再檢查一次，避免舊歷史紀錄或其他事件直接跳轉繞過規則；可仿照「在咖啡廳工作」開頭。

## 開放時間、開鎖與警衛規則

若場所白天可正常進入、關門後可嘗試開鎖，請在同一個場所資料加入 `access`。大學是完整範例；開鎖共用流程在 `src/VenueAccess.twee`，通常不需要修改。

```js
access: {
    entryPassage: "大學",
    returnPassage: "市區",
    returnLabel: "返回市區",
    openDays: [1, 2, 3, 4, 5],
    openHour: 7,
    closeHour: 19,
    lockpick: { difficulty: 15, /* 其他數字請複製大學範例 */ }
}
```

- 星期日是 `0`、星期一到星期五是 `1`～`5`、星期六是 `6`。
- 入口按鈕使用 `<<venueEnter "🏫大學" "university" 3>>`；最後的 `3` 是前往該場所花的分鐘數。
- 開鎖成功率依「基礎值 + 靈巧等級加成 + 有效幸運加成 - 難度」計算；每個場所可設定自己的 `difficulty`。
- 開鎖失敗但未被警衛發現時，該場所冷卻 60 分鐘；被發現時扣聲望，並只封鎖該場所到當天結束。
- `successMessage`、`failedMessage`、`caughtMessage`、`lockoutMessage` 與 `cooldownMessage` 都能依場所改成專屬事件文字。

## 角色性別與大學科系

新遊戲按下開始後，會先選擇 `$playerGender`，再選擇 `$majorId`；確認頁的「我再想一下」可回到選系畫面。可編輯選項集中在 `setup.gameDatabase.characterGenders` 與 `setup.gameDatabase.majors`。

每個科系只要指定一個核心 `majorSkill`；玩家之後可把另一個科系加選為副修：

```js
example_major: {
    name: "範例科系",
    majorSkill: "tech",
    description: "這裡是選系畫面的簡短介紹。"
}
```

大學課程與畢業後講師工作的數值都在 `venueRules.university.education`。目前學生每堂課為 60 分鐘、每 14 個遊戲日繳 1,200 元學費、每次選定的主修或副修課 +12 XP。學費不足時學生不能上課；可至學生服務中心補足後恢復。

新開局只選主修，之後可在學生服務中心加選一個不同科系為副修。主修滿級，且未選副修或副修也滿級時會畢業；畢業後科系改顯示為學位，不能再當學生上課，但可教授自己取得學位的科系。

入學首 7 天可免費轉主修；逾期每次收 1,600 元。這兩個數值分別是 `majorChangeFreeDays` 與 `majorChangeFee`。轉系不會重設技能，舊主修則保留在 `$educationHistory`，但不再能繼續修讀。

講師數值集中在 `education.lecturer`：目前每堂 60 分鐘、Lv.1 薪資 80 元、每級加 15 元、每堂 +20 講師經驗；所有數值都可直接調整，無須修改故事段落。

## 公園休息與角色資訊

- 公園休息的分鐘數與恢復量在 `setup.balance.parkRest`。目前為 15 分鐘、體力 +1、心情 +2；它仍會正常推進時間，所以不會是無代價的恢復。
- 左側的「角色」按鈕會開啟獨立角色資訊視窗，顯示性別、科系與文字化聲望。聲望的分段文字集中在 `setup.balance.reputation.tiers`，不需要到介面檔案修改。
- 日後的搶劫、被搶、約會、親密事件與第一次事件紀錄，應先在 `StoryInit` 建立統計變數，再在 `skills.twee` 的 `CharacterSheet` 區塊下方顯示；這樣事件程式與介面會保持分開。
- 左側的五項常駐數值使用 `<<statusBar>>` 共用元件顯示為緊湊進度條：體力／心情為第一列，飢餓／衛生為第二列，慾望獨佔最後一行；金錢與幸運另放在下方。一般數值在 25% 以下變黃、10% 以下變紅；慾望固定為紫色。若日後新增同類數值，直接複製 `StoryCaption` 的一行並換成對應名稱、目前值、最大值與類型即可。
- 進度條與金錢／幸運包在 `status-panel`；其上方預留的頭像空位由 `_UserStyle.twee` 的 `margin-top: 165px` 控制。新增玩家頭像時，可依實際圖片高度調整這個數字。低數值不再顯示額外文字警告，直接以黃／紅進度條提示，讓左側版面保持緊湊。

## 新增物品範例

在 `locales` 的每一種語言各加入 `items.cookie` 的 `name`、`description`、`useMessage`，再加入以下資料：

```js
cookie: {
    icon: "🍪",
    category: "food",
    tags: ["consumable", "food", "shop_711", "gift"],
    price: 35,
    stackable: true,
    usable: true,
    effects: [{ type: "stat", stat: "satiety", amount: 10 }]
}
```

若要在 7-11 販售，再將 `cookie` 加到 `shops.shop_711.itemIds`。背包、購物介面、結帳金額與購物車會自動支援它。

## NPC 資料設計

每個 NPC 有固定資料（`tags`、生日、喜好、事件 ID）與存檔資料（`$npcRelations`）兩部分。不可戀愛角色只有 `affection`；可戀愛角色另有 `romance`、`relationshipStatus` 與 `romancePrompted`。

可戀愛 NPC 起初和其他角色相同，使用好感度。好感第一次達到 `romanceThreshold`（預設 80）時，系統會在目前事件結束後顯示共用關係確認；接受後好感的數值會轉入愛意，狀態改為 `romance`。分手劇情可用 `<<run setup.requestBreakup("npcId")>><<goto "NPC_分手確認">>` 開始；愛意會依 `breakupAffectionPenalty`（預設 40）扣減後轉回好感。

事件中請一律使用 `<<run setup.changeNpcAffection("npcId", 數值)>>` 改好感；交往後的愛意改用 `<<run setup.changeNpcRomance("npcId", 數值)>>`。這兩個函式都會將數值限制在 -100～100。

`alice` 與 `bob` 是依目前規劃文件建立的暫定範例，不代表已設定的正式角色。`alice` 示範可戀愛、忠誠、會在關係事件中敏感反應的角色；`bob` 示範不可戀愛角色。角色設定完成時，直接保留 ID 並替換翻譯、標籤與資料即可。

## TAG 與條件範例

TAG 可同時用於分類和規則。現有 `setup.conditionMet()` 支援以下條件：

```js
{ type: "npcHasTag", npcId: "alice", tag: "loyal" }
{ type: "npcLacksTag", npcId: "bob", tag: "romanceable" }
{ type: "hasItem", itemId: "tea", quantity: 1 }
{ type: "itemHasTag", itemId: "tea", tag: "gift" }
{ type: "relationshipAtLeast", npcId: "alice", stat: "romance", value: 80 }
{ type: "flag", flag: "alice_saw_public_flirting" }
```

日後的事件可先檢查這些條件，再套用結果。例如「具有 `loyal` 標籤的戀人目睹公開曖昧」可扣除其 `romance`；當數值低於角色的 `breakupAtRomance`（目前範例為 40）時，再觸發分手事件。多重交往不需要特別限制，因為每位角色的關係數值是分開儲存的。
