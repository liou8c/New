# 遊戲狀態與平衡規格

> 目的：這份文件是「會影響遊戲規則與存檔」的共同說明。新增系統、事件或數值時，先更新這裡，再改程式。
>
> 不記錄只在單一 passage 內使用的 `_暫用變數`，例如 `_roll`、`_workPay`、`_cafeOrderGreeting`；它們離開該 passage 後不應作為遊戲狀態使用。

## 先用白話理解這份文件

你可以把遊戲想成一間店：

- **遊戲狀態**是「這位玩家現在的狀況」：有多少錢、幾點鐘、體力多少、認識誰、劇情做到哪裡。每一位玩家的存檔都不同。
- **遊戲規則**是「店的規定」：每小時扣多少飽食、房租多少、工作時薪多少、搶匪出現的機率。所有玩家共用同一套。
- **物品／NPC 資料**是「商品與角色名錄」：泡麵賣多少、紅茶回多少心情、溫莉叫什麼名字。它們在資料庫中集中管理。

所以，以後你想做的事與該改的位置如下：

| 你想調整的事 | 優先去看哪裡 |
|---|---|
| 讓遊戲整體更難／更簡單 | `src/_GameDatabase.twee` 最上方的 `setup.balance` |
| 改新開局金錢、體力或數值上限 | `setup.balance.player` |
| 改房租、時間流逝、工作、健身房、睡眠、搶匪 | `setup.balance` 裡對應名稱的區塊 |
| 改一個物品的價格或效果 | `setup.gameDatabase.items` |
| 改 NPC 名稱、頭像、關係初值 | `setup.gameDatabase.npcs` 與 `locales.npcs` |
| 新增故事事件的進度條件 | `$flags` 與 NPC／事件規格 |

**初學者守則：**如果看到 `$money`、`$hp`，那是玩家正在遊玩的資料，通常不要在設定檔直接修改；如果看到 `setup.balance`，才是你用來調整整體規則的地方。

## 維護規則

1. `$` 開頭的是 SugarCube 的故事變數，會寫入玩家存檔。
2. 固定遊戲資料放在 `setup.gameDatabase`；玩家目前擁有的數量、關係或進度放在 `$` 變數。
3. 每個新變數都要有：預設值、唯一或主要修改入口、可接受範圍，以及舊存檔缺少它時的預設行為。
4. 事件旗標統一使用 `$flags["小寫_id"]`，例如 `$flags["winley_park_intro_done"]`。不要混用 `$flags.xxx` 寫法。
5. 時間只能由 `<<passTime ...>>`／`window.advanceTime()` 推進；不要在一般事件中直接修改 `$gameDate`。
6. 要改全遊戲共用數值（上限、消耗率、價格、薪水、機率）時，先在「待集中平衡值」找到或建立唯一設定位置；不要複製數字到多個 passage。

## 重要資料結構

### `setup.gameDatabase`（固定資料，不放進存檔）

檔案：`src/_GameDatabase.twee`

| 區塊 | 用途 | 維護原則 |
|---|---|---|
| `locales` | 物品、NPC 的顯示文字 | key 必須與資料 ID 對齊；新增語言時保留同一組 ID |
| `items` | 物品圖示、價格、標籤、效果 | 新增物品後，初始化與商店會依 ID 建立庫存／購物車欄位 |
| `npcs` | NPC 顏色、標籤、關係初值、社交資料 | NPC ID 不要任意改名，否則舊存檔的關係資料會失聯 |
| `locations` | 地點 ID、passage、標籤 | 現在與 `PassageHeader` 的 tag 對應並存；未來應選一處作為唯一來源 |
| `shops` | 商店販售的 item ID 清單 | 商品必須存在於 `items` |

### `TimedEffect`

目前由限時道具建立，欄位包括：`itemId`、`stat`、`amount`、`originalValue`、`expireAmount`、`debuffDurationMinutes`、`phase`、`expiresAt`。

- `phase` 為 `"buff"` 時，`expiresAt` 到期會轉為 `"debuff"`。
- `phase` 為 `"debuff"` 時，到期後回復使用前的數值並移除效果。
- 新增限時道具時，應先確認「同類效果能否疊加」與「使用期間改變該數值時如何結算」。

## 待集中平衡值（下一步整理目標）

這些不是現在就要調到完美，而是要在下一次程式整理時移到少數設定位置：

1. 所有基本數值的初始值與最大值。
2. 每小時的飽食、體力、衛生、慾望變化。
3. 房租週期與金額。
4. 工作薪水與工作消耗。
5. 健身房、餐點與物品價格／效果。
6. 事件機率、冷卻時間、成功率公式。
7. 技能升級所需經驗、等級上限與升級獎勵。

建議建立 `setup.balance`（固定設定）並由所有 passage 讀取它。例如不再在故事段落直接寫 `100`、`600`、`120` 或 `50`。

## 存檔相容性規則

目前資料庫有 `setup.gameDatabase.version`，但玩家存檔尚未有正式版本。新增會保存的狀態時，應執行以下流程：

1. 在初始化處給新變數明確預設值。
2. 在載入既有存檔時，若變數不存在，補上同一個預設值。
3. 對 ID 改名採「保留舊 ID 或轉換資料」策略；不要直接改字串。
4. 每次會破壞舊存檔的改動，都提高存檔版本並記錄轉換原因。

目前已完成：`$robberEventCooldownUntil`、`$faintEventActive`、`$faintedAtHome` 的正式初始化，以及 `setup.ensureGameState()` 的統一舊存檔補值。之後新增存檔欄位時，照同一模式補上即可。

## 新增內容前的小檢查

- 新事件是否有唯一 ID、觸發條件、完成旗標／階段、結果與回歸地點？
- 新選項是否透過 `<<passTime>>` 結算應流逝的時間？
- 新物品是否同時補上資料庫定義、顯示文字、販售位置與效果？
- 新 NPC 是否同時補上資料庫資料、關係初值、認識時機與社交頁資訊？
- 新增的數值改動是否使用正確上限，而非直接寫死 `100`？
- 新增的存檔變數是否已寫入本文件，並考慮舊存檔？
